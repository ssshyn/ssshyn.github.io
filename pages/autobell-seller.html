프로젝트 개요

현대글로비스 오토벨 셀러 플랫폼 고도화 프로젝트에 참여하여, 중고차 판매자를 위한 거래 프로세스를 통합 관리할 수 있도록 서비스 전반을 개선하였습니다.


기존 수출 플랫폼 고도화가 관리자 중심의 기능 개선이었다면, 본 프로젝트는 차량을 직접 등록하고 판매하는 셀러 사용자의 이용 흐름을 중심으로 사용자 경험을 개선하는 데 목적이 있었습니다.


차량 매물 등록부터 보험 및 차량 이력 조회, 주문, 결제 완료까지 이어지는 거래 전 과정을 하나의 흐름으로 통합하여 관리할 수 있도록 도메인 로직을 재구성하였으며, 이를 통해 단계별로 분리되어 있던 프로세스를 일관된 구조로 개선하였습니다.


또한, 거래 상태 변화에 따른 실시간 알림을 제공하기 위해 FCM 기반 푸시 알림 기능을 도입하였으며, 운영 환경에서 Git 및 Jenkins 서버 변경에 대응하여 배포 프로세스를 안정적으로 유지하는 데 기여하였습니다.

주요 업무

  • 차량 매물 등록부터 주문 및 결제 완료까지 이어지는 거래 단계 도메인 설계 및 API 개발
  • 거래 상태 변화에 따른 알림 기능의 안정성을 확보하기 위해 FCM 푸시 설정 및 발송 로직을 개선
  • Git/Jenkins 환경 변경에 따른 배포 프로세스 대응 및 Feature 브랜치 기반 운영 안정화 지원
  • 배포 시 브랜치 머지 및 Jenkins 반영 관리로 안정적인 릴리즈 운영 기여

트러블슈팅

관리자-셀러 간 차량/주문 상태값 불일치로 인한 도메인 정합성 문제 해결
Problem

차량 및 주문 프로세스 진행 과정에서 관리자 시스템과 셀러 시스템 간 상태값 정의가 상이하여, 동일한 데이터임에도 각 시스템에서 서로 다른 단계로 인식되는 문제가 발생하였습니다. 이로 인해 차량 조회 시 실제 진행 상태를 정확하게 파악하기 어려운 상황이었습니다.

Cause

기획 및 설계 단계에서 관리자 기준의 프로세스와 셀러 시스템의 상태 정의가 통일되지 않은 채 개발이 진행되었습니다. 그 결과 동일한 업무 흐름임에도 시스템별로 서로 다른 상태값을 사용하게 되었고, 도메인 정합성이 깨진 상태였습니다. 또한 상태값에 대한 공통 기준과 매핑 정의가 부재했던 것이 주요 원인이었습니다.

Solve

관리자 및 셀러 시스템에서 사용 중인 전체 상태값을 정리하여 공유하였으며, 실제 업무 처리 기준을 기반으로 상태 간 매핑 기준을 수립하도록 협의하였습니다. 이후 차량 조회 시 관리자와 셀러의 상태를 함께 확인할 수 있도록 내부 매핑 로직을 구현하였으며, 이를 통해 두 시스템 간 상태를 일관되게 해석하고 동기화할 수 있도록 개선하였습니다.

회고

이번 프로젝트에서는 기획 의도와 실제 기획서 간의 불일치로 인해 개발 진행 중 기능 변경 요청이 빈번하게 발생하는 상황을 경험하였습니다. 이를 통해 초기 기획 단계에서 개발 관점의 검토와 커뮤니케이션이 충분히 이루어지지 않을 경우, 프로젝트 일정과 개발 효율에 큰 영향을 줄 수 있음을 체감하였습니다.


또한, 기존 알림 및 거래 관련 소스 코드 분석 과정에서 불필요하거나 정리되지 않은 로직이 다수 존재하여 기능 이해 및 수정에 예상보다 많은 시간이 소요되었습니다. 이를 통해 유지보수성을 고려한 코드 구조 설계와 지속적인 리팩토링의 중요성을 다시 한번 인식하게 되었습니다.


오픈 이후 안정화 단계에서는 개발 PL의 부재로 인해 운영팀 및 현업 담당자와 직접 커뮤니케이션하며 결함 처리 및 기능 개선을 수행하였습니다. 이 과정에서 요구사항을 보다 명확히 이해하고 빠르게 대응할 수 있었으며, 업무에 대한 몰입도와 책임감이 높아지는 경험을 하였습니다.


향후에는 초기 기획 단계에서 개발 관점의 리스크를 보다 적극적으로 공유하고, 변경 가능성을 고려한 유연한 구조 설계와 함께 최소한의 리팩토링 기준을 유지하며 개발을 진행하고자 합니다.
