삼성증권 TAX 플랫폼 구축

애드캡슐소프트

2024.12 — 2025.05 Java6/8 Spring Boot JEUS8 Swagger

프로젝트 개요

삼성증권의 TAX 플랫폼 구축 프로젝트에 참여하여, 고객의 금융 및 비금융 자산을 기반으로 소득과 세액을 통합 조회하고 절세 전략 수립을 지원하는 서비스를 구현하였습니다.

해당 플랫폼은 국세청 및 정부24 데이터를 연계하여 연간 소득 및 세액을 조회하고, 증권사 내 자산과 타사 자산을 통합 분석하여 세금 리포트 및 절세 시뮬레이션을 제공하는 것을 목표로 하였습니다.

본 프로젝트에서 저는 백엔드 개발자로 참여하여 외부 스크래핑 데이터를 기반으로 내부 시스템과 연계되는 API를 개발하고, 계산된 세금 데이터를 가공하여 클라이언트에 제공하는 역할을 수행하였습니다.

특히, 외부 스크래핑 업체와 내부 시스템 간 데이터 흐름을 정리하고 요청/응답 구조를 표준화하여 연계 안정성을 높였으며, 다양한 이해관계자(상위 기획사, 내부 기획자, 시스템 담당자) 간 커뮤니케이션을 통해 요구사항을 조율하고 API 명세를 구체화하는 데 기여하였습니다.

또한, 기존 웹 환경이 Java6, Ant 기반의 레거시 구조로 구성되어 있어 현대적인 애플리케이션 환경과의 차이를 이해하고 연계하는 과정에서 레거시 시스템에 대한 적응력과 유지보수 경험을 함께 확보하였습니다.

이를 통해 단순 기능 구현을 넘어, 복잡한 시스템 간 데이터 흐름을 안정적으로 연결하고 서비스 품질을 향상시키는 데 기여하였습니다.

주요 업무

  • REST API 개발 및 프론트 연동 지원
  • 국세청/정부24 스크래핑 데이터 연동을 위한 인터페이스 정의 및 데이터 처리 로직 구현
  • JEUS 로컬 환경 구축 및 ANT 빌드 설정을 통해 레거시 시스템 개발 환경 구성
  • 기획자 및 외부 업체와의 기술 커뮤니케이션을 통해 요구사항 구체화 및 API 명세 정립

트러블슈팅

스크래핑 연동 API 테스트 데이터 확보 제약
Problem

국세청/정부24 스크래핑 데이터 특성상 실제 사용자 기반 데이터에 의존해야 했기 때문에 테스트 데이터 확보에 제약이 있었습니다.

Solve

이를 해결하기 위해 테스트용 성명/주민번호 입력 시 사전에 정의한 JSON 기반 Mock 데이터를 반환하도록 구성하여, 외부 의존 없이도 API 기능을 검증할 수 있는 환경을 구축하였습니다.


이후 실제 테스트 단계에서는 고객 협조를 통해 다양한 실데이터를 확보하였고, 그 과정에서 발생한 누락 데이터 및 예외 케이스를 식별하여 데이터 처리 로직을 보완하였습니다.


이를 통해 외부 데이터 의존성이 높은 환경에서도 안정적인 서비스 동작을 확보할 수 있었습니다.

외부 스크래핑 API 응답 불일치 대응 및 데이터 안정성 개선
Problem

외부 스크래핑 솔루션 연동 과정에서 필드명이 잘못 전달되거나 응답 코드가 명세와 다르게 내려오는 문제가 발생하였고, 이로 인해 API 파싱 오류 및 기능 정상 동작에 문제가 발생하였습니다.

Cause

외부 솔루션 API의 응답 스펙이 명확히 고정되지 않은 상태에서 개발이 진행되었으며 일부 케이스에서 필드명 오타, 응답 코드 불일치 등 명세와 실제 응답 간 차이가 발생하였습니다.

Solve

문제 발생 시 응답 로그를 기반으로 원인을 분석하고, 외부 업체와 협의하여 정확한 스펙을 재정의하였습니다.

또한, 변경된 응답 구조에 맞춰 파싱 로직 및 DTO를 수정하여 정상적으로 데이터를 처리할 수 있도록 개선하였습니다. 이를 통해 외부 API 변경에도 유연하게 대응할 수 있는 구조를 확보하였습니다.

Maven 버전 변경으로 인한 Nexus 의존성 다운로드 실패 해결
Problem

IntelliJ 최신 버전 환경에서 프로젝트 빌드 시 Nexus Repository를 통한 의존성 다운로드 및 일부 Maven Plugin resolve가 실패하는 문제가 발생하였습니다.

Cause

IntelliJ에 번들된 Maven 버전이 상향되면서 HTTP 기반 Repository 접근이 기본적으로 차단됨에 따라 기존 Nexus 저장소와의 통신이 정상적으로 이루어지지 않았습니다.

또한, 일부 Maven Plugin의 경우 pom.xml에 버전이 명시되지 않아 Maven의 기본 Plugin 버전 해석 과정에서 resolve 오류가 발생하였습니다.

Solve

Maven settings.xml에 mirror 설정을 추가하여 HTTP Repository 접근이 가능하도록 우회 경로를 구성하고 Nexus를 통한 의존성 다운로드가 정상적으로 이루어지도록 개선하였습니다.

Plugin resolve 오류는 pom.xml에 명시적인 버전을 지정하여 의존성 해석을 명확히 함으로써 해결하였습니다. 해당 설정을 팀원들과 공유하여 동일한 개발 환경 이슈가 재발하지 않도록 표준화하였습니다.

회고

외부 스크래핑 솔루션과 내부 시스템에 의존하는 구조에서 개발을 진행하면서, 개발자가 모든 로직과 데이터를 직접 통제할 수 없는 환경을 경험하였습니다.


특히, 외부 API의 응답 스펙 변경이나 데이터 편차로 인해 예상하지 못한 문제가 발생하는 상황을 겪으며, 변화에 유연하게 대응할 수 있는 구조 설계와 예외 처리의 중요성을 깊이 이해하게 되었습니다.


또한, 실사용자 기반 데이터에 의존하는 특성상 테스트 데이터 확보에 제약이 있었고, 이를 해결하기 위해 Mock 데이터 구성 및 시나리오 기반 검증을 수행하였습니다.


이후 실제 데이터 검증 과정에서 다양한 누락 데이터와 예외 케이스를 경험하면서 단순 기능 구현을 넘어, 테스트 전략을 사전에 설계하는 것이 서비스 안정성에 직결된다는 점을 체감하였습니다.


이를 통해 외부 의존성이 높은 환경에서는 유연한 대응 구조와 체계적인 테스트 전략이 필수적이라는 점을 배울 수 있었습니다.
